# ZedUploader 🤖📁

ربات آپلودر فایل تلگرام با معماری تولیدی (Production-ready) مبتنی بر
Python · aiogram 3 · FastAPI · PostgreSQL · Redis · Docker.

---

## معرفی پروژه

**ZedUploader** یک ربات تلگرامی است که به ادمین‌ها اجازه می‌دهد فایل/رسانه آپلود
کنند. ربات به‌جای ذخیرهٔ خود فایل، شناسهٔ `file_id` تلگرام را به‌همراه متادیتا در
PostgreSQL نگه می‌دارد، برای هر رسانه یک **کد کوتاه یکتا** می‌سازد، و کاربر از طریق
یک **لینک اختصاصی (deep link)** فایل را دریافت می‌کند:

```
https://t.me/BOT_USERNAME?start=MEDIA_CODE
```

## امکانات MVP (فاز ۱)

- 📤 آپلود فایل توسط ادمین‌ها (عکس، ویدیو، فایل، صوت، ویس، انیمیشن/گیف، استیکر)
- 🔗 تولید کد کوتاه یکتا و لینک اختصاصی برای هر رسانه
- 🔒 قفل فوروارد (`protect_content`) به‌صورت اختیاری
- ⏳ حذف خودکار زمان‌دار پیام‌ها با صف پایدار Redis + ورکر مستقل
  (پس از ری‌استارت هم از بین نمی‌رود)
- 🎯 محدودیت تعداد دانلود به‌صورت **اتمیک و بدون Race**
- 🧾 ثبت لاگ دانلودها
- 🌐 API فقط-خواندنی محافظت‌شده با `X-API-Key` و محدودیت نرخ (rate limit)
- 🧱 اجرای کامل با Docker Compose (api / bot / worker / db / redis)

## پیش‌نیازها

- سرور Ubuntu (۲۰.۰۴ یا جدیدتر)
- دامنه‌ای که به IP سرور اشاره کند (برای وب‌هوک و SSL)
- توکن ربات از [@BotFather](https://t.me/BotFather)
- Docker و Docker Compose v2 (اسکریپت نصب در صورت نبود، آن‌ها را نصب می‌کند)

## نصب روی Ubuntu (تک‌دستوری)

```bash
git clone https://github.com/Mhoseinshah1/zed-uploader && cd zed-uploader && sudo bash install.sh
```

> ⚠️ **مهم:** همیشه پروژه را با `install.sh` اجرا کنید. این اسکریپت اسرار واقعی
> (`WEBHOOK_SECRET`، `API_KEY`، `JWT_SECRET`) و رمز پایگاه‌داده را تولید می‌کند.
> اگر مستقیماً `docker compose up` بزنید، مقادیر پیش‌فرض و ناامن `change_this_*`
> و رمز پیش‌فرض دیتابیس استفاده می‌شود که برای محیط عملیاتی خطرناک است.

اسکریپت `install.sh` این کارها را انجام می‌دهد:

1. نصب بسته‌های پایه (curl، git، openssl) و در صورت نیاز Docker
2. ساخت فایل `.env` از روی `.env.example`
3. پرسیدن حالت ربات: **polling** (بدون نیاز به دامنه/SSL، مناسب تست) یا
   **webhook** (عملیاتی، نیازمند دامنه)
4. پرسیدن `BOT_TOKEN`، `BOT_USERNAME`، `ADMIN_IDS` (و `DOMAIN` فقط در حالت webhook)
5. تولید خودکار `WEBHOOK_SECRET`، `API_KEY`، `JWT_SECRET` و رمز پایگاه‌داده
   (و همگام نگه‌داشتن `DATABASE_URL` با `POSTGRES_PASSWORD`)
6. ساخت ایمیج‌ها، اجرای دیتابیس/ردیس، اجرای مایگریشن‌ها و بالا آوردن سرویس‌ها
7. در حالت **webhook**: نصب و پیکربندی خودکار nginx + گواهی Let's Encrypt و سپس
   تنظیم وب‌هوک تلگرام. در حالت **polling** ربات بلافاصله کار می‌کند.

## تنظیم دامنه و SSL (nginx + certbot)

در حالت **webhook**، اسکریپت `install.sh` این مراحل را به‌صورت خودکار انجام می‌دهد.
بخش زیر برای انجام دستی یا رفع اشکال است. فایل نمونهٔ `nginx/uploader-bot.conf`
ترافیک را به `127.0.0.1:8000` (uvicorn با `--proxy-headers`) می‌رساند و برای
certbot آماده است:

```bash
sudo cp nginx/uploader-bot.conf /etc/nginx/sites-available/uploader-bot.conf
sudo ln -s /etc/nginx/sites-available/uploader-bot.conf /etc/nginx/sites-enabled/
# مقدار server_name را به دامنهٔ خود تغییر دهید
sudo nginx -t && sudo systemctl reload nginx

sudo apt install -y certbot python3-certbot-nginx
sudo certbot --nginx -d your.domain
```

پس از گرفتن گواهی SSL، مقدار `DOMAIN` در `.env` باید `https://your.domain` باشد و
وب‌هوک را دوباره تنظیم کنید (اجرای مجدد `install.sh` یا فراخوانی دستی `setWebhook`).

3. توکن داده‌شده را در `BOT_TOKEN` قرار دهید و یوزرنیم را در `BOT_USERNAME`.

به [@userinfobot](https://t.me/userinfobot) پیام دهید تا عدد شناسهٔ عددی شما را

## اجرا

```bash
docker compose up -d           # اجرای همهٔ سرویس‌ها
docker compose ps              # وضعیت سرویس‌ها
```

سرویس‌ها:

| سرویس   | دستور                         | توضیح                         |
|---------|-------------------------------|-------------------------------|
| `api`   | `uvicorn app.api.main:app`    | API و دریافت وب‌هوک تلگرام    |
| `bot`   | `python -m app.bot.main`      | polling یا ثبت وب‌هوک         |
| `worker`| `python -m app.workers.main`  | حذف خودکار زمان‌دار پیام‌ها   |
| `db`    | postgres 16                   | پایگاه‌داده                   |
| `redis` | redis 7                       | صف حذف خودکار + rate limit    |

## مشاهده لاگ‌ها

```bash
docker compose logs -f            # همهٔ سرویس‌ها
docker compose logs -f worker     # فقط ورکر حذف خودکار
docker compose logs -f api bot    # api و bot
```

## اجرای migration

```bash
docker compose run --rm api alembic upgrade head     # اعمال آخرین نسخه
docker compose run --rm api alembic downgrade -1     # بازگشت یک نسخه
```

## به‌روزرسانی

برای دریافت نسخهٔ جدید و اعمال همهٔ تغییرات به‌صورت خودکار و **بدون از دست رفتن
داده‌ها**، فقط این دستور را اجرا کنید (خود اسکریپت کد را می‌کشد؛ نیازی به
`git pull` دستی نیست):

```bash
cd ~/zed-uploader && sudo bash update.sh
```

اسکریپت `update.sh` به‌صورت کاملاً خودکار (بدون پرسش):

1. **قبل از هر تغییر** از دیتابیس یک پشتیبان می‌گیرد در
   `‎/backups/pre-update-<timestamp>.sql` و فقط ۷ نسخهٔ آخر را نگه می‌دارد.
2. کد را با `git pull --ff-only` می‌کشد و ایمیج‌ها را بازسازی می‌کند.
3. `db` و `redis` را بالا آورده و منتظر سالم‌شدنشان می‌ماند، سپس مایگریشن‌ها را
   اعمال می‌کند. **اگر مایگریشن شکست بخورد، اسکریپت متوقف می‌شود** و مسیر فایل
   پشتیبان را برای بازیابی نمایش می‌دهد (سرویس‌ها روی نسخهٔ قبلی می‌مانند).
4. سرویس‌ها را با ایمیج جدید بالا می‌آورد و سلامت API را با
   `‎http://127.0.0.1:8000/health` بررسی می‌کند.
5. تغییرات نسخهٔ قبلی تا نسخهٔ جدید را (`git log --oneline OLD..NEW`) چاپ می‌کند.

این اسکریپت هرگز `docker compose down -v` نمی‌زند، به `pgdata` و فایل `.env` دست
نمی‌زند، و اجرای مکرر آن امن است.

> پنل مدیریت (دکمه‌های ادمین) پس از به‌روزرسانی به‌صورت خودکار فعال می‌شود؛ کافی
> است در تلگرام `/start` یا `/panel` را بزنید.

## پشتیبان‌گیری و بازیابی

از پنل وب (بخش «پشتیبان») می‌توانید پشتیبان دستی بگیرید، زمان‌بندی روزانه/هفتگی
تنظیم کنید (نگه‌داری N نسخهٔ آخر)، فایل‌ها را دانلود یا حذف کنید.

> **⚠️ بازیابی (Restore) مخرب است:** دیتابیس فعلی را کاملاً با فایل پشتیبان
> جایگزین می‌کند و قابل بازگشت نیست. فقط مالک باید انجامش دهد، قبل از آن یک
> پشتیبان تازه بگیرید، و برای تأیید باید نام دقیق فایل را تایپ کنید. دامپ‌ها با
> `--clean --if-exists` گرفته می‌شوند تا روی دیتابیس موجود قابل اعمال باشند.

## خطاهای رایج

- **ربات پاسخ نمی‌دهد:** مقدار `BOT_MODE` را بررسی کنید. در حالت `webhook` باید
  `DOMAIN` روی HTTPS معتبر تنظیم و وب‌هوک ست شده باشد؛ برای تست محلی می‌توانید
  `BOT_MODE=polling` بگذارید.
- **خطای اتصال به دیتابیس:** مطمئن شوید سرویس `db` سالم (`healthy`) است و
  `DATABASE_URL` با `POSTGRES_*` هماهنگ است.
- **وب‌هوک ست نمی‌شود:** `DOMAIN` باید در دسترس عمومی و دارای SSL معتبر باشد.
- **حذف خودکار کار نمی‌کند:** لاگ سرویس `worker` و سلامت `redis` را بررسی کنید.

## پنل مدیریت (فاز ۲)

- **آپلود گروهی (📚):** با دکمهٔ «آپلود گروهی» چند فایل را یکی‌یکی بفرست و سپس
  «پایان» را بزن تا همه زیر **یک لینک** قرار گیرند. (گروه‌بندی خودکار آلبوم
  کشیدن‌ورها کردن در نقشهٔ راه است.)
- **عضویت اجباری (🔐):** می‌توانید کاربران را ملزم کنید پیش از دریافت فایل در
  کانال‌ها عضو شوند. ⚠️ **این قابلیت فقط زمانی کار می‌کند که ربات در هر کانال
  اجباری، ادمین باشد** (در غیر این‌صورت آن کانال نادیده گرفته می‌شود / fail-open).
- **مدیریت ادمین‌ها (👥، مخصوص مالک):** افزودن/حذف/غیرفعال‌سازی ادمین‌ها از داخل
  ربات. مالک‌های تعریف‌شده در `ADMIN_IDS` قابل حذف نیستند.
- **پیام همگانی (📢، مخصوص مالک):** ارسال پیام به همهٔ کاربران؛ ارسال توسط
  سرویس `worker` انجام می‌شود تا ربات بلاک نشود، و کاربران بلاک‌کننده علامت
  می‌خورند. (پردازش at-least-once؛ در صورت کرش ممکن است چند ارسال تکراری رخ دهد.)

## کیف پول و اشتراک (فاز ۳)

- **کیف پول (💳):** موجودی + «افزایش موجودی» (کارت‌به‌کارت) و «تراکنش‌ها». موجودی
  فقط توسط `WalletService` و همراه با ثبت ردیف دفترکل تغییر می‌کند؛ تأیید پرداخت
  **idempotent** است (تأیید دوباره، دوبار شارژ نمی‌کند).
- **اشتراک (⭐️):** خرید پلن‌های `plus`/`max` از موجودی کیف پول؛ قابلیت‌های
  محافظت محتوا / حذف خودکار / آپلود گروهی و سقف تعداد فایل بر اساس پلن گیت می‌شوند.
- **فروش (💼، مخصوص مالک):** تنظیم شمارهٔ کارت و نام صاحب کارت و قیمت/مدت هر پلن.
  تا وقتی شمارهٔ کارت تنظیم نشده باشد، افزایش موجودی غیرفعال است.
- مالک‌های تعریف‌شده در `ADMIN_IDS` از همهٔ محدودیت‌های پلن معاف‌اند.

## پنل وب مدیریت (فاز ۴)

یک داشبورد وب امن (RTL/فارسی) روی همان سرویس‌های ربات، زیر مسیر `/panel` که توسط
سرویس `api` سرو می‌شود.

- **ساخت کاربر پنل:** `install.sh` هنگام نصب نام کاربری و رمز عبور را می‌پرسد و
  کاربر پنل را می‌سازد. برای ساخت/تغییر دستی:
  ```bash
  docker compose run --rm api python -m app.panel.create_user --username U --password P
  ```
- **دسترسی:**
  - عملیاتی (با دامنه و SSL): `https://<domain>/panel`
  - محلی/بدون TLS: چون `api` فقط روی `127.0.0.1:8000` باز است، از تونل SSH استفاده کنید:
    ```bash
    ssh -L 8000:127.0.0.1:8000 root@server
    # سپس در مرورگر: http://localhost:8000/panel
    ```
    هرگز پورت api را بدون TLS به‌صورت عمومی باز نکنید.
- **امنیت:** رمزها با bcrypt، نشست‌های سمت‌سرور در Redis با کوکی امضاشده
  (`HttpOnly`, `SameSite=Lax`, و `Secure` روی HTTPS)، توکن CSRF برای هر فرم،
  محدودیت نرخ ورود (قفل پس از چند تلاش ناموفق)، هدرهای امنیتی (CSP سخت‌گیرانه،
  `X-Frame-Options`)، و ثبت audit برای هر اقدام. تمام دارایی‌ها محلی‌اند (بدون CDN).
- **صفحات:** داشبورد، کاربران (مسدودسازی/تعدیل موجودی از طریق `WalletService`)،
  فایل‌ها، پرداخت‌ها (تأیید/رد idempotent)، پلن‌ها، ادمین‌ها، تنظیمات، پیام همگانی.

> نکته: منطق تجاری (نوشتن موجودی فقط توسط `WalletService`، تأیید idempotent پرداخت)
> در سرویس‌ها می‌ماند؛ پنل فقط یک لایهٔ نازک UI است.

## درگاه CentralPay (فاز ۵)

پرداخت آنلاین در کنار کارت‌به‌کارت، هم برای افزایش موجودی و هم برای خرید پلن.

- **فعال‌سازی:** دو کلید را در `.env` قرار دهید (یا هنگام نصب وارد کنید):
  `CENTRALPAY_GETLINK_KEY` (کلید ساده) و `CENTRALPAY_VERIFY_KEY` (کلید md5).
  اگر هر کدام خالی باشد، گزینهٔ «پرداخت آنلاین» خودکار پنهان و درگاه غیرفعال
  می‌شود و فقط کارت‌به‌کارت کار می‌کند.
- **returnUrl:** درگاه کاربر را با GET به
  `https://<DOMAIN>/pay/centralpay/return?orderId=<id>` برمی‌گرداند؛ این آدرس باید
  از طریق nginx و روی HTTPS در دسترس باشد. `DOMAIN` همان مقدار نصب است.
- **پیش‌نیاز:** DNS دامنه به این سرور اشاره کند، گواهی معتبر Let's Encrypt نصب
  باشد، و مسیر بالا از طریق nginx برسد. تا قبل از آن، درگاه کار نمی‌کند.
- **تست سلامت مسیر:**
  ```bash
  curl -i https://<DOMAIN>/pay/centralpay/return?orderId=0
  # باید به اپ برسد (صفحهٔ «پرداخت ناموفق»)، نه خطای nginx/certbot
  ```
- **امنیت پول:** چون IPN امضاشده وجود ندارد، تأیید پرداخت **idempotent** (قفل
  ردیف + بررسی وضعیت) و با **تطبیق مبلغ/کاربر** انجام می‌شود؛ هرگز سفارشِ
  قبلاً‌تأییدشده دوباره تأیید/شارژ نمی‌شود و در صورت مغایرت هیچ شارژی انجام
  نمی‌شود. ردیف‌های CentralPay در پنل فقط‌خواندنی‌اند و تأیید دستی ندارند
  (فقط ردیف‌های `card` دستی تأیید می‌شوند).

## چند-ربات، پشتیبانی و فاکتور (فاز F–H)

- **چند-مستأجری:** یک نصب می‌تواند چندین ربات مشتری را میزبانی کند؛ هر ربات
  کاملاً ایزوله است (داده/پنل/لاگ/بکاپ جداگانه). فروش ربات فقط از «ربات اصلی».
  جزئیات: `docs/multitenant.md`.
- **پشتیبانی (تیکت):** دکمهٔ «🎧 پشتیبانی» در هر ربات. کاربر تیکت به ادمینِ همان
  ربات می‌زند؛ نماینده (ادمین) می‌تواند تیکت به «پلتفرم» بزند که فقط در پنل
  سوپرادمین دیده می‌شود. پاسخ از پنل، اطلاع‌رسانی به کاربر.
- **فاکتور/رسید:** برای هر پرداختِ تسویه‌شده (شارژ، خرید پلن، ساخت/تمدید ربات)
  دقیقاً یک فاکتور (idempotent) ساخته می‌شود؛ رسید در ربات و لیست فاکتورها در
  پنل با خروجی CSV.

## آماده‌سازی تولید (فاز I)

- **مسدودسازی واقعی کاربر:** کاربر مسدود در هیچ مسیری (هندلر، لینک عمیق، پیام
  همگانی) سرویس نمی‌گیرد؛ ادمین/اونر مستثنا هستند.
- **نقش‌های پنل:** هر کاربرِ پنل یک نقش دارد — `owner | admin | support |
  finance | content`. دسترسی‌ها: کیف‌پول/پرداخت‌ها → owner/finance؛
  رسانه/بازبینی → owner/admin/content؛ تنظیمات/درگاه‌ها/پلن‌ها/تیم/قابلیت‌ها →
  owner؛ مشاهدهٔ کاربران → همهٔ نقش‌ها (به‌جز content). سوپرادمین پلتفرم مستقل است.
  مدیریت نقش‌ها در «کاربران پنل» (owner).
- **مدیریت دستی کیف‌پول/اشتراک:** در `/panel/users/{id}` شارژ/کسر دستی با «علت»
  الزامی + تأیید مبالغ بزرگ + reference به‌صورت `panel:<panel_user_id>` +
  اطلاع‌رسانی به کاربر؛ و تغییر پلن/تمدید/دائمی/لغو اشتراک — همه ممیزی‌شده.
- **تلگرام استارز (کلید کلی):** روشن/خاموش برای هر ربات از صفحهٔ درگاه‌ها؛ در
  حالت خاموش دکمه پنهان و `pre_checkout` رد می‌شود.
- **پنل رسانهٔ کامل + Feature Flags:** ویرایش عنوان/کپشن/سقف دانلود/حذف خودکار/
  وضعیت + صفر‌کردن شمارنده + لینک عمیقِ ربات همان مستأجر؛ جستجو با
  کد/عنوان/کپشن/نام فایل. صفحهٔ «قابلیت‌ها» (owner) برای فعال/غیرفعال و حداقل پلن.
- **پیام همگانی و پرداخت:** فیلتر مخاطب (همه/رایگان/پولی؛ مسدودها هرگز)، کلید
  مستقلِ کارت‌به‌کارت، حداقل مبلغ شارژ، فیلتر و صفحهٔ جزئیات پرداخت، و «بازبینی
  درگاه» (idempotent؛ تأیید دستی فقط کارت‌به‌کارت).
- **webhook و استارز:** `install.sh` مقدار
  `allowed_updates=["message","callback_query","pre_checkout_query"]` را تنظیم
  می‌کند (نبودِ `pre_checkout_query` باعث شکست بی‌صدای پرداخت استارز می‌شود).

## نقشه راه (Phase 6+)

معماری برای افزودن موارد زیر باز است (بدون پیاده‌سازی در این نسخه):

- **درگاه‌های بیشتر:** زرین‌پال و کریپتو (فیلد `payments.method` و seam نازک
  `PaymentProvider` از پیش لحاظ شده).
- **آلبوم native:** گروه‌بندی خودکار چند فایلِ کشیده‌شده در یک آپلود.
- **پنل:** ورود دومرحله‌ای (Telegram-OTP / 2FA) به‌عنوان seam مستند.
